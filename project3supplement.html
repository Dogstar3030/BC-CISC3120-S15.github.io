Project 3: Supplemental Suggestions

Now that you've had some time to ponder project 3 (and, hopefully, get the checkers game running in proper MVC style--make sure you have an Observable and an Observer involved!), here are a few thoughts about how to make the transition to the networked version. Remember that, unlike the Chatter-related lab/projects, here the client and server have very different roles: the server 'hosts' a game which will be played by two clients (and, possibly, observed by other clients).

First, think about making the client as minimal as possible. The client should incorporate the view and the controller: what's the least amount of behavior it could reasonably have? It needs to be able to display the current state of a checkers game, of course, and it should probably be smart enough to know whose turn it is (so it can prevent the user from trying to take a turn at the wrong time).

You might also want to give the client the ability to decide whether the user is making a legal move, or you could give it the ability to eliminate obviously illegal moves (like moving a checker non-diagonally). That's up to you--though the most minimal client wouldn't even be able to do that. It's OK to have the server evaluate potential moves, too--it might not be the most efficient approach, but it's fine for the purposes of this project.

Then: what information, exactly, does the client need in order to be able to do its job? Hint: it probably does not need a reference to CheckerBoard object. How far can you minimize the client's information needs?

Once you've decided exactly what the client will do, think about the kind of communication the clients and server will need. What will the client need from the server? What the client have to send to the server? Don't worry quite as much about minimialism here, but of course the more information you send back and forth, the potentially slower the game will be.

As usual, the impact of your effort will be greatest if you use as much existing code as you can. Once you get the networked game working, then you can experiment with adding chat, improving the UI, etc.

Now that I've discovered that the Oracle tutorial has been rewritten to use JSON instead of XML (so there's no helpful code for you to study), I should point out a couple key features of any class you want to encode or decode using JAXB.

First, the class should be annotated with @XmlRootElement (and you'll also need to import javax.xml.bind.annotation.* ). So, for example, you might have

import javax.xml.bind.annotation.*;
@XmlRootElement
class MoveMessage {

And so on.

Second, your class needs to be composed entirely of "getters and setters." That is, the only constructor you have should be the default (no parameters) constructor, and for each instance variable x , you should have a getX() and setX() method. This is very easy to do in Eclipse: once you have your instance variables listed, just highlight that set of lines, then under the Source menu, select Generate Getters and Setters. Eclipse will write the proper methods.
